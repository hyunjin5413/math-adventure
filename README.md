# 수학 어드벤처 — 커리큘럼 & 문제 상세 설계

PRD v0.1의 **§4(커리큘럼) · §6(문제 구조) · §12(콘텐츠 파이프라인)** 를 실제로 동작하는
데이터 + 생성기로 구체화한 산출물입니다. 100스테이지 × 약 20문제 = **1,975문제**를
시드 기반으로 재현 가능하게 자동 생성하고, 정답·보기·범위를 자동 검수합니다.

```
src/skills.mjs       개념(스킬) 27종: 문제 생성기 + 오답 보기(distractor) 규칙
src/curriculum.mjs   100스테이지 마스터 정의 (5월드 × 20, 미니복습/보스 자동 생성)
src/generate.mjs     생성 + 검증 + 통계 CLI
schema/stage.schema.json   비개발자 저작용 JSON Schema (PRD §11)
output/stages.json   생성된 전체 콘텐츠 (검수용 정답 포함)
output/summary.json  스킬/입력/월드 분포 + 검증 리포트
```

## 실행

```bash
node src/generate.mjs                 # 전체 생성 → output/
node src/generate.mjs --variants 3    # 스테이지별 변형 3벌 (매 시도 다른 문제, §12)
node src/generate.mjs --stage 15      # 스테이지 1개 미리보기(stdout)
```

## 설계 개념: 스킬 = 개념 원자

PRD §4.1 "한 스테이지 = 한 핵심 개념"을 지키기 위해 학습 개념을 27개 **스킬**으로 쪼갰습니다.
각 스킬은 ① 파라미터화된 문제 생성기와 ② 교육적으로 의미 있는 **오답 보기 규칙**을 가집니다.

오답 보기는 무작위가 아니라 *대표적 오개념*을 노립니다(진단 가치):

| 스킬 | 정답 외 대표 오답 | 노리는 오개념 |
|---|---|---|
| `add_carry` (받아올림) | `(a+b) % 10` | 받아올림을 빼먹음 |
| `sub_borrow` (받아내림) | `|일의자리 - 빼는수|` | 큰 수에서 작은 수로 거꾸로 뺌 |
| `multiplication` (구구단) | `정답 ± 한 단`, `a×(b±1)` | 단 건너뛰기 / 인접 곱 혼동 |
| `place_value` (자리값) | 십·일 자리 뒤집기 | 자리값 미형성 |
| `number_recognition` | 6↔9, 2↔5 등 | 시각적으로 닮은 숫자 혼동 |

## 100스테이지 흐름 (5월드 × 20)

각 월드 10번째 = 미니 복습(직전 9스테이지 혼합), 20번째 = 보스전(월드 종합 20문제).
일반 스테이지는 3번째부터 직전 개념을 **20% 간격 반복**으로 섞습니다(PRD §4.1-4).

| 월드 | 무드 | Phase | 핵심 진행 |
|---|---|---|---|
| W1 공룡 알 마을 | 공룡 | 0~1 | 수세기 → 비교 → (그림→숫자식) 합10이하 덧셈 → 10만들기/모으기 |
| W2 변신 정비소 | 변신 로봇 | 1~2 | 세수 덧셈 → 가르기 → 뺄셈 → 10에서 빼기 → 빈칸/수가족 |
| W3 특공대 기지 | 특공대 | 3 | 10채우기 전략 → 받아올림(9+,8+,…) → 받아내림 → 혼합 |
| W4 히어로 시티 | 히어로 | 4~5 | 자리값 → 두자리 ±  → 뛰어/묶어 세기 → 반복덧셈 → 배 개념 |
| W5 몬스터 도감 | 수집형 몬스터 | 6 | 배열 의미 → 2·5단 → 3·4단 → 6~9단 → 빈칸/문장제 → 종합 |

CRA 원칙(구체→표상→추상)은 같은 개념을 입력 방식으로 단계화해 구현합니다:
`tap_count`(사물 세기) → `choice`/`drag_match`(표상) → `keypad`(숫자식 추상).

## 입력 유형 분포 (PRD §6.2)

생성 결과: `keypad 914 · choice 716 · fill_blank 176 · tap_count 105 · drag_match 64`.
어린 Phase일수록 보기 수를 줄이고(2~3개) 후반은 3~4개로 자동 조절합니다.

## 검수 파이프라인 (PRD §12)

- **결정적 시드**: `seed = n*1000 + variant*7 + 13` → 같은 입력은 항상 같은 문제. 재현·디프 가능.
- **자동 검증**(`output/summary.json`):
  1. 정답 재계산 일치 (operands/operator로 독립 재연산)
  2. 보기에 정답 포함 & 중복 없음
  3. 음수/비현실 답 차단
- 현재 상태: **오류 0건 / 1,975문제**.

## 프론트엔드 연동 메모

- 문제는 렌더링과 분리된 순수 데이터입니다. `visual.type`이 그릴 컴포넌트를 지정합니다:
  `objects`, `ten_frame(_add/_sub)`, `compose_bond`, `base_ten(_add/_sub)`, `number_line`,
  `groups`, `array`, `compare_groups`, `spoken_number`.
- `hintRef`는 §6.3 적응형 힌트(연속 오답 2회 시 노출)에 매핑됩니다:
  `make_ten`, `count_on`, `count_back`, `subtract_to_ten`, `array`, `number_line`, `base_ten` 등.
- `prompt.tts`는 읽기 보조 음성(§8.3)용. 숫자는 한자어 수사로 읽습니다(`numKo`).

## 플레이 가능한 프로토타입 (web/)

생성한 `output/stages.json`을 그대로 렌더링하는 React 프로토타입입니다. 빌드 도구 없이
ESM CDN(React 18) + `htm`으로 구성해 정적 서버만 있으면 바로 실행됩니다.

```bash
node src/generate.mjs          # 먼저 콘텐츠 생성
python3 -m http.server 8137    # 프로젝트 루트에서 정적 서버
# 브라우저에서 http://localhost:8137/web/ 접속 (태블릿 가로 권장)
```

구현 범위(PRD 반영):
- **월드 맵**: 5월드 × 20스테이지, 잠금/해제(클리어 시 다음 해제), 별 표시, 보스(👑)·미니복습(🔁) 구분
- **플레이**: 진행 바, 콤보, 점수, 사물/텐프레임/배열/자리값/수직선 등 `visual.type`별 그림
- **입력**: 보기 선택 / 숫자 키패드 (drag·tap·fill은 각각 보기·키패드로 매핑)
- **점수·별점**(§7): 기본 +10, 속도/콤보 보너스, 정답률 기반 1~3★, 결과 화면(총점/콤보/정답률/보상)
- **무감점 재도전**(§4.1-6): 오답 시 같은 문제 재시도, 연속 오답 2회 시 힌트 노출(§6.3)
- **TTS 읽기 보조**(§8.3): Web Speech API로 문제/피드백 음성 안내
- **PWA 뼈대**: manifest + 가로 모드 안내. (서비스워커/오프라인 캐시는 미구현 — 다음 단계)
- 진척도는 `localStorage`에 저장.

검증: 정적 서버에서 맵→플레이→정답 입력→다음 문제 진행까지 동작 확인(콘솔 에러 0).

## PRD 부록과의 정합성

- 스테이지 15(10 만들기), 92(구구단), 보스전(60) 모두 PRD 부록 예시와 형식이 일치합니다.
- 단, 부록 A의 "S92 = 3단" 예시는 단을 쉬운→어려운 순으로 재배치하며 9단으로 조정했고
  3단은 S85에 배치했습니다(아래 "확인 필요" 참조).

## 확인이 필요한 결정 사항

1. **스테이지당 문제 수**: 현재 일반/보스 20, 미니복습 15. PRD "약 20"에 맞췄으나 7세 집중시간
   (약 10분, §3.1)을 고려하면 12~15가 적정일 수 있음 → 베타에서 보정 권장.
2. **곱셈 도입 시점**: 두 자리 뺄셈(받아내림 포함)은 W4에서 가볍게만 다루고 W5 전체를 곱셈에
   할애. 두 자리 받아내림을 더 강화할지 결정 필요.
3. **단 순서**: 부록 A의 3단(S92) 예시 vs 본 설계(난이도순). 어느 쪽을 정본으로 할지.
4. **드래그 매칭 비중**: 현재 64문제로 가장 적음. 수 감각/곱셈 준비에서 더 늘릴지.
